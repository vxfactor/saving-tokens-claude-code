How Tokens Actually Work (And Why You're Bleeding Them)

Most people think of Claude Code's context window like a hard drive. You put stuff in, it stays there, and you use it when you need it.

Wrong mental model. The context window is active working memory — more like a whiteboard than a filing cabinet. Everything on it gets processed every single turn. Every file Claude reads, every tool call result, every response it generates — all of it stays on the whiteboard and gets re-read on the next turn.

This is why your sessions slow down and quality degrades. It's not that Claude "forgets." It's that the whiteboard is so full of old notes that it can't focus on what matters right now.

Where Your Tokens Actually Go

Here's what most people don't realize about token consumption:

  • Input tokens are charged every turn for the entire conversation history. Turn 1 costs X. Turn 20 costs 20X, because Claude re-reads everything.
  • Tool call results are often the biggest culprits. A single file read can dump 2,000+ lines into context. That stays there for every subsequent turn.
  • Claude's own responses also count. A verbose response on turn 3 inflates every turn after it.
  • CLAUDE.md and rules files are loaded once at the start — but they're in context for the entire session.

The practical consequence: a 50-turn session doesn't cost 50x a single turn. It costs roughly **50 * (average context size)**, which grows with every turn.

The Four Failure Modes

When context gets bloated, you'll see these symptoms in order:

  1. Inconsistent code — Claude contradicts decisions from earlier in the session
  2. Repeated questions — it asks about project structure you already explained
  3. Lost conventions — naming patterns and architectural decisions disappear
  4. Breaking changes — it ignores patterns established minutes ago

If you've hit any of these, you weren't doing anything wrong. Your context window was just full.

The 80% Rule

Watch the token percentage in Claude Code's status bar. When you hit ~80%, you have two options:

  1. Run /compact to compress the conversation history
  2. Exit and start a fresh session for complex multi-file work
Don't wait until quality degrades to act. By the time you notice bad output, you've already wasted tokens on responses you'll need to redo. Check the status bar proactively.

The rest of this course is about never hitting 80% in the first place — or at least, not hitting it until you've gotten 3-5x more useful work done per session.

CLAUDE.md: The File That Saves You Thousands of Tokens Per Session

Every time you start a new Claude Code session and explain your project setup, your coding preferences, your API configurations, your deployment patterns — that's tokens you're burning on re-explanation.

CLAUDE.md is the fix. It's a markdown file at your project root that Claude reads automatically at the start of every session. Think of it as persistent memory between conversations.

But most people's CLAUDE.md files are either empty or filled with vague instructions like "write clean code." Here's how to make yours actually save tokens.

The Structure That Works

markdown
# Project Name

One-line description of what this project does.

## Tech Stack
- Language, framework, key dependencies
- Database, hosting, CI/CD

## Key Paths
| Path | Purpose |
|------|------|
| src/api/ | API routes |
| src/models/ | Database models |
| .env | API keys (never commit) |

## Capabilities
*Living log. After meaningful work, append what worked.*
- Deploy script handles zero-downtime via blue-green swap
- Image pipeline uses Wavespeed API, not local generation

## What Not To Do
*Living log. After failures, append what went wrong.*
- Don't use the v1 endpoint — deprecated, returns 500s
- Don't mock the database in integration tests
- Airtable image URLs expire — use thumbnails.full.url

The Two Sections That Save the Most Tokens

Capabilities and What Not To Do are living logs. After every meaningful session, you append one line about what worked or what failed.

Over time, these compound. Instead of Claude rediscovering that "the v1 API endpoint is broken" (costing you 10 minutes and hundreds of tokens in failed attempts), it reads it in CLAUDE.md on session start and skips straight to the working approach.

Prompt
After completing a task, add to CLAUDE.md: "Append what worked to Capabilities, what failed to What Not To Do. One line each."

Hardcode Known Values

This is the single highest-ROI trick I use. If there's a value Claude looks up every time — an API ID, a config setting, a deployment URL — hardcode it in your CLAUDE.md or skill docs.

markdown
## Constants
- Typefully social set ID: 288354
- Airtable content base: appXXXXXXXX
- Deploy target: production-east-1
- CTA link: https://tidycal.com/nicsilver/intro-call

Every hardcoded value eliminates a lookup. A lookup means a tool call, which means tokens for the request and the response, which stays in context for the rest of the session. One hardcoded line saves 200+ tokens per session.

Rules Files: Auto-Loading Behavior

CLAUDE.md isn't the only file Claude reads automatically. Files in .claude/rules/ are loaded every session too. Use these for cross-cutting standards:

text
.claude/rules/
  code-style.md    → "Use snake_case, keep functions under 20 lines"
  testing.md       → "Integration tests hit real DB, unit tests mock external APIs only"
  git.md           → "Conventional commits, never force-push main"

The key insight: rules files replace repeated instructions. If you find yourself typing the same guidance across sessions, it belongs in a rules file. Write it once, never explain it again.

Keep your CLAUDE.md under 200 lines. Dense, not comprehensive. If it gets too long, you're putting context-window pressure on every session. Move reference material to separate files that Claude reads on demand.

Model Tiering: Stop Using Opus for Everything

If you're running Opus for every task, you're using a sledgehammer to hang picture frames. Opus is the most capable model, but it's also the most expensive in tokens-per-turn. For tasks that don't need its full reasoning power, you're burning tokens for nothing.

Model tiering means matching the model to the task complexity. Claude Code makes this easy — you can set model overrides per skill, per subagent, or per session.

The Three Tiers

TierModelUse WhenToken Cost
Research & BreadthSonnetSearching code, reading files, exploring options, gathering dataLow
Synthesis & JudgmentOpusFinal output, complex reasoning, architectural decisions, merging researchHigh
Classification & LookupsHaikuYes/no decisions, categorization, simple extraction, filteringVery Low

Concrete Examples

Use Sonnet for:

  • Scanning a codebase for all files matching a pattern
  • Reading and summarizing documentation
  • First-pass research across multiple sources
  • Generating multiple draft options (where you'll pick the best one anyway)
  • Running exploration subagents that fan out across the codebase

Use Opus for:

  • Writing the final implementation after research is done
  • Making architectural trade-off decisions
  • Synthesizing findings from multiple Sonnet agents into a coherent plan
  • Complex refactoring where correctness matters more than speed
  • The main conversation thread where you're orchestrating work

Use Haiku for:

  • Filtering a list of items by relevance ("is this file related to authentication?")
  • Simple data extraction ("pull the email from this JSON")
  • Pre-screening before handing to a more expensive model
  • Classification tasks ("is this a bug report or a feature request?")

How to Set It

In skill definitions (.claude/skills//SKILL.md):

yaml
---
model: sonnet  # This skill runs on Sonnet
---

In subagent definitions (.claude/agents/.md):

yaml
---
model: haiku   # This agent runs on Haiku
---

When spawning agents inline:

Prompt
"Use Sonnet subagents for the research phase, then synthesize with Opus in the main thread."

The Fan-Out Pattern

The most token-efficient pattern I use daily:

  1. Fan out 3-5 Sonnet agents in parallel, each researching one angle
  2. Collect their findings (they report back concisely)
  3. Synthesize in the main Opus thread — one turn of high-quality reasoning

Instead of Opus doing 15 turns of research + synthesis, you get Sonnet doing 15 turns of cheap research + Opus doing 1 turn of expensive synthesis. The total token cost drops dramatically, and the output quality is often better because each agent has a focused, clean context.

Don't overthink tier selection. When in doubt, start with Sonnet. If the output quality isn't good enough, bump to Opus. You'll quickly develop intuition for which tasks need which tier.

Subagents and Parallel Execution

Subagents are the most powerful token-saving mechanism in Claude Code — and the most underused. Every subagent runs in its own isolated context window. That means it only sees what you explicitly pass to it, not your entire 50-turn conversation history.

context: fork — The Key Setting

When you define a skill or agent with context: fork, it runs in a completely isolated context. It doesn't inherit your conversation history. It gets a clean slate with just the instructions you give it.

yaml
---
name: research-agent
description: Researches a topic across the codebase
model: sonnet
context: fork
---

You are a research agent. Given a topic, search the codebase
for all relevant files and return a summary.

Input: $ARGUMENTS

Without context: fork, the subagent inherits your full conversation — which might be 100K tokens of unrelated context. With it, the subagent starts with maybe 2K tokens of instructions. That's a 50x reduction in input tokens per turn.

Scripts Do the Work, Claude Orchestrates

This is the pattern that saves the most tokens in my daily workflow: don't use Claude for deterministic work.

If a task has a predictable input/output — an API call, a data transformation, a file format conversion — write a Python script for it. Claude's job is to decide *when* to run the script and *what* to do with the results.

text
# Bad: Claude makes the API call (tokens for request + response + reasoning)
"Call the Airtable API to get all records from the content table,
 filter by status=draft, and return the titles."

# Good: Script makes the call, Claude gets just the result
python3 scripts/get_drafts.py --status draft --fields title

The script approach has three benefits:

  1. Fewer tokens — Claude sees a 5-line result instead of reasoning through API docs
  2. Deterministic — the script works the same way every time, no prompt variance
  3. Reusable — the script runs without Claude, useful for cron jobs and CI

Parallel Agent Pattern

Instead of one agent doing 5 tasks sequentially (each task adding to the context), spawn 5 agents that each do 1 task in parallel:

yaml
# .claude/agents/wiki-scanner.md
---
name: wiki-scanner
model: sonnet
context: fork
---
Scan the wiki index for pages related to: $ARGUMENTS
Return a bullet list of relevant page paths and one-line summaries.
yaml
# .claude/agents/trend-scanner.md
---
name: trend-scanner
model: sonnet
context: fork
---
Search for trending content on the given topic.
Return top 5 findings with source URLs.

Then in your main skill:

Prompt
Fan out 3 parallel sonnet agents in a single message: wiki-scanner, trend-scanner, and content-inventory. Each searches independently. Synthesize their results in the main thread.

Each agent runs with a tiny context window (~5K tokens). The main thread receives three concise summaries instead of accumulating 50K tokens of sequential exploration.

The Practical Difference

ApproachContext per turnTotal tokens (10 tasks)
One agent, sequentialGrows: 10K → 50K → 100K~500K+
Parallel agents, forkedFixed: ~5K each~50K total
Scripts + orchestrationFixed: ~2K per script result~20K total

That's not a marginal improvement. That's the difference between hitting your limit at lunch and making it through the full day.

Subagents can't spawn other subagents. If you need chained delegation, chain from the main conversation thread. Plan your agent hierarchy before you start.

When NOT to Optimize (The Counter-Intuitive Section)

Everything in the previous sections can make your token usage dramatically more efficient. But there's a trap: premature token optimization kills flow and can actually cost you more.

I learned this the hard way. I spent a week over-constraining my CLAUDE.md, limiting tool access, forcing minimal responses. The result? Claude produced worse output that needed redoing. Every redo cost more tokens than the original "wasteful" approach would have.

When Spending More Tokens Saves Tokens

Exploration phases. When you don't know what you're looking for, let Claude explore. Restricting file reads and searches to "save tokens" means Claude guesses instead of looking. Guesses lead to wrong approaches. Wrong approaches lead to debugging sessions that burn 10x more tokens than the exploration would have.

Complex reasoning. Sometimes you need Opus to think through a problem in a long, detailed response. Forcing terse responses on complex architectural decisions leads to shallow thinking, which leads to flawed implementations, which leads to costly rewrites.

First drafts. Don't optimize your prompts to death before seeing what Claude produces. A slightly wasteful first attempt that works is infinitely cheaper than a perfectly optimized prompt that produces the wrong thing.

The Real Cost Equation

The cost of a Claude Code session isn't just tokens consumed. It's:

Total cost = tokens consumed + (tokens wasted on bad output that needs redoing) + (your time spent re-prompting)

A session that uses 200K tokens and produces the right result on the first try is cheaper than a session that uses 80K tokens but needs three attempts because you over-constrained it.

Five Signs You're Over-Optimizing

  1. You're writing longer prompts to save tokens. If your instructions about how to be efficient are themselves consuming significant tokens, you've gone too far.
  2. Claude keeps asking clarifying questions. You removed too much context. Now it's burning tokens on back-and-forth instead.
  3. Output quality dropped. Terse CLAUDE.md instructions produce terse, shallow output. Dense doesn't mean cryptic.
  4. You're spending more time managing Claude than using it. If you're micro-managing model selection and context for every minor task, the management overhead exceeds the savings.
  5. You're avoiding useful tools. Skipping file reads or web searches to "save tokens" means Claude works with incomplete information.

The Rules I Follow

  • Optimize systems, not individual prompts. CLAUDE.md, rules files, skills, and subagent architecture are system-level optimizations. They compound. Micro-optimizing a single prompt is a one-time saving that rarely compounds.
  • Let exploration be expensive. Research and discovery phases should be unconstrained. Optimization belongs in the execution phase, after you know what you're building.
  • Measure by outcome, not by tokens. A 300K-token session that ships a feature is better than three 80K-token sessions that each fail halfway.
  • Optimize when you feel the pain. If you're consistently hitting limits before your work is done, then optimize. If you're not, don't fix what isn't broken.
The best token optimization is getting the right answer on the first try. Invest in clear instructions, good context, and appropriate model selection — not in restricting Claude's ability to do its job.
